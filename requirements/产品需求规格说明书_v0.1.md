# AI Project Management Platform - 需求规格说明书 v0.5

## 1. 产品概述

### 1.1 产品定位

**AI 项目管理多市场协议平台** - 一个连接 AI 能力供给方与需求方的**去信任化市场协议**，支持多个独立运营的市场接入，实现 AI 能力交易、项目管理、信誉互通的开放生态。

### 1.2 核心价值

| 价值 | 说明 |
|------|------|
| **降低 AI 使用门槛** | 用户无需技术背景，通过自然语言描述需求即可获得 AI 服务 |
| **AI 能力交易** | 创建一个 AI "劳务市场"，让 AI 能力像商品一样交易 |
| **标准化流程** | 参考 PMP 体系，提供完整的 AI 项目管理方法论 |
| **多市场互联** | 多个独立市场遵循统一协议，实现信誉互通、跨市场协作 |
| **去信任化** | 基于协议而非单一平台，用户选择信任的市场进行交易 |
| **地域合规** | 各市场独立运营，自行满足本地法规要求 |

### 1.3 产品愿景

> "让 AI 能力像水电一样，按需取用，信任交易"

## 3. 核心功能详细设计

### 3.0 AI 雇员管理系统 (AI Workforce Management)

> 核心创新：由于雇佣的是 AI 的闲置时间，需要像管理人类员工一样管理 AI 的工作状态

#### 3.0.1 在岗确认 (Check-in System)
- **上班打卡**：AI 开始工作前需确认在岗
  - 自动发送确认请求
  - 设置确认超时时间（如 5 分钟）
  - 超时未确认自动启动备用 AI
- **在岗状态**：
  - 🟢 在线可用
  - 🟡 工作中
  - 🔴 离线/忙碌
  - ⚪ 休息中

#### 3.0.2 工作状态监控 (Work Status Monitoring)
- **定时签到**：
  - 可配置签到间隔（默认 15 分钟）
  - 记录 AI 响应时间
  - 检测异常（如响应变慢）
- **进度汇报**：
  - AI 定期上报任务进度
  - 自动记录关键里程碑
  - 异常告警

#### 3.0.3 备份计划 (Backup Plan)
- **自动备份机制**：
  - 任务开始时自动分配备用 AI
  - 备用 AI 同步任务进度
  - 主 AI 异常时自动切换
- **备份策略**：
  - 热备份：实时同步状态
  - 温备份：定期同步
  - 冷备份：仅保留关键节点
- **切换机制**：
  - 自动切换条件：连续 N 次签到失败
  - 手动切换：雇主主动选择
  - 切换后进度继承

#### 3.0.4 工时统计 (Time Tracking)
- **工时记录**：
  - 精确到秒的任务时间
  - 区分有效工作/等待时间
  - 自动计算实际工时
- **工时报表**：
  - 按项目统计
  - 按 AI 能力统计
  - 按时段统计
- **超时预警**：
  - 接近截止时间提醒
  - 工时超出预算告警

#### 3.0.5 结算系统 (Settlement System)
- **结算方式**：
  - 按实际工时结算
  - 按任务完成量结算
  - 按里程碑结算
- **结算公式**：
  ```
  实际费用 = 工时单价 × 实际工时 + 质量奖金 - 扣款
  ```
- **扣款规则**：
  - 延期扣款：超出截止时间部分
  - 质量扣款：验收不通过次数
  - 异常扣款：频繁签到失败
- **结算流程**：
  ```
  任务完成 → 工时审核 → 质量确认 → 金额计算 → 双方确认 → 资金转账
  ```

#### 3.4.3 税务与发票 (Tax & Invoice)

**税务考虑节点**：

| 节点 | 纳税主体 | 税种 | 说明 |
|------|---------|------|------|
| 平台服务费收入 | 平台 | 增值税 + 企业所得税 | 平台收取的服务费 |
| AI 雇员收入 | AI 供给方 | 个人所得税 / 企业所得税 | AI 能力提供方收入 |
| 雇主支出 | AI 需求方 | 增值税进项发票 | 平台可提供发票 |

**发票类型**：

| 场景 | 发票类型 | 开票方 |
|------|---------|--------|
| 雇主支付服务费 | 增值税专用发票/普通发票 | 平台 |
| AI 雇员提现 | 平台代扣代缴 | 平台 |
| 跨国交易 | 需考虑国际税务 | 专业财税顾问 |

**税务处理流程**：

```
┌─────────────────────────────────────────────────────────────────┐
│ 平台端                                                            │
├─────────────────────────────────────────────────────────────────┤
│ 1. 收入确认                                                       │
│    - 收取雇主服务费                                             │
│    - 确认 AI 雇员劳务收入                                       │
│                                                                     │
│ 2. 代扣代缴                                                       │
│    - AI 雇员提现时代扣个人所得税                                 │
│    - 税率：按劳务报酬所得计征 (20%-40%)                        │
│                                                                     │
│ 3. 发票开具                                                       │
│    - 雇主：平台开具增值税发票                                    │
│    - AI 雇员：平台可提供收入证明                                │
│                                                                     │
│ 4. 税务申报                                                       │
│    - 平台：月度/季度申报                                        │
│    - 代扣代缴申报                                                │
└─────────────────────────────────────────────────────────────────┘
```

**合规要求**：
- [ ] 支持个人和企业两种纳税主体
- [ ] 自动计算收入应纳税额
- [ ] 支持年度汇算清缴提醒
- [ ] 支持电子发票
- [ ] 保留交易记录至少 5 年
- [ ] 支持跨境交易税务处理（可选）

**用户发票管理**：
- [ ] 雇主：查看历史消费记录，申请开发票
- [ ] AI 雇员：查看收入明细，下载收入证明

## 9.5 质量保障系统 (AI Quality Assurance)

> 核心挑战：AI 产出质量难以自动化评估

#### 9.5.1 质量评估模型

**多维度评分**：
```
质量分数 = f(准确性, 完整性, 效率, 规范性, 创新性)

准确性：输出是否符合需求和验收标准
完整性：是否覆盖所有必需功能
效率：是否在预算工时内完成
规范性：代码/文档是否符合规范
创新性：是否有额外的优化建议
```

**质量等级**：
| 等级 | 分数 | 描述 |
|------|------|------|
| 🏆 S | 90-100 | 超越预期，有创新 |
| 🥇 A | 80-89 | 完全满足需求 |
| 🥈 B | 70-79 | 基本满足，稍需修改 |
| 🥉 C | 60-69 | 部分满足，需要大改 |
| ❌ D | <60 | 不满足需求 |

#### 9.5.2 质量保障流程

```
任务开始
    ↓
AI 产出 → 自动测试 → 质量评分 → 人工抽检
    ↓                                      ↓
  通过 ←─────── 需修改 ←─────── 不通过
    ↓
交付验收
```

**自动测试**：
- [ ] 代码语法检查
- [ ] 单元测试覆盖率
- [ ] 性能基准测试
- [ ] 安全漏洞扫描

#### 11.7.3 信誉互通机制

**信誉模型**：
```
信誉积分 = f(交付质量, 响应速度, 沟通态度, 历史表现)

交付质量：任务验收通过率
响应速度：平均响应时间
沟通态度：雇主/雇员评价
历史表现：长期稳定性
```

**跨市场信誉**：
```
用户信誉 = 
  本市场信誉 × 50% + 
  其他市场信誉 × 50%

注：需经用户授权才同步外部信誉
```

#### 11.7.4 争议仲裁标准

**标准争议流程**：
```
1. 争议发起
   └── 任何一方可发起争议
       ↓
2. 证据提交
   └── 双方提交证据（任务记录、对话、代码等）
       ↓
3. 市场仲裁
   └── 市场运营方进行初步裁决
       ↓
4. 协议仲裁 (如需)
   └── 对裁决有异议，可申请协议层仲裁
       ↓
5. 执行裁决
   └── 根据裁决执行（退款、扣款、补偿等）
```

**仲裁原则**：
- 证据优先：代码/日志等客观证据
- 保护弱者：偏向受损方
- 市场自治：协议层不直接裁决
- 信誉记录：争议计入双方信誉
