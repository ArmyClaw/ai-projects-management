# AI Project Management - 迭代开发Skill

> 基于设计文档进行迭代式开发，测试驱动，Clean Code

## 核心原则

```
├─ 测试驱动开发（TDD）
├─ Clean Code + 详细注释
├─ 每15分钟小迭代
├─ 每3小时需求校准
└─ 迭代记录沉淀
```

---

## 开发流程

### Step 1: 需求读取

```
1. 读取设计文档
   ├─ design/产品需求规格说明书_v0.8.md
   ├─ design/架构设计文档_v0.8.md
   ├─ design/原型设计文档_v0.8.md
   └─ design/场景设计文档_v0.8.md

2. 读取开发规范
   └─ develop/README.md

3. 读取进度
   └─ progress/README.md

4. 读取上次迭代记录
   └─ progress/iterations/v0.x-YYYY-MM-DD.md
```

### Step 2: 任务拆解

```
基于需求拆解任务：
├─ 功能点
├─ 对应文件
├─ 测试用例
└─ 预期工时

优先级：
P0: 核心功能
P1: 重要功能
P2: 次要功能
```

### Step 3: 测试驱动开发

```
开发流程：
1. 编写UT（测试用例）
2. 运行UT（失败）
3. 编写功能代码
4. 运行UT（通过）
5. 编写注释
6. 运行全部UT
7. 代码Review
8. 提交代码
9. 推送到远程仓库 ✅
```

### Step 4: Clean Code标准

```
命名规范：
├─ 文件：PascalCase（Vue组件）
├─ 函数：camelCase
├─ 常量：UPPER_SNAKE_CASE
└─ 类型：PascalCase

注释要求：
├─ 函数：JSDoc/TSDoc
├─ 复杂逻辑：行内注释
├─ 业务规则：说明背景
└─ TODO：标注待办

代码质量：
├─ 单函数 < 50行
├─ 单文件 < 300行
├─ 圈复杂度 < 10
└─ 重复代码 < 3处
```

### Step 5: 迭代记录

```
每次迭代完成：
├─ 记录文件：progress/iterations/v0.x-YYYY-MM-DD-HHmm.md
├─ 包含：
│  ├─ 迭代时间
│  ├─ 完成功能
│  ├─ 代码变更
│  ├─ 测试结果
│  ├─ 问题与解决
│  └─ 待决策项
└─ 更新进度表：progress/README.md
```

## 需求校准（每3小时）

```
校准清单：
├─ 当前开发 vs 设计文档
│   ├─ 功能完整性
│   ├─ API一致性
│   └─ 数据模型一致性
├─ 代码质量
│   ├─ UT覆盖率
│   ├─ 注释完整性
│   └─ Clean Code遵循度
├─ 进度偏差
│   ├─ 已完成 vs 计划
│   ├─ 阻塞项
│   └─ 风险项
└─ 决策项
    ├─ 可自行决策 → 执行
    └─ 无法决策 → 记录+通知
```

---

## 决策原则

### 可自行决策项

```
├─ 代码规范细节
├─ 函数拆分粒度
├─ 变量命名优化
├─ 注释补充
├─ 小型重构
├─ 测试用例补充
└─ 工具库版本更新

决策依据：
├─ 符合设计文档意图
├─ 符合Clean Code标准
├─ 不影响现有功能
├─ 不改变API契约
└─ 有利于后续维护
```

### 必须记录项

```
├─ 数据模型变更
├─ API接口变更
├─ 核心逻辑变更
├─ 功能增删
├─ 依赖库重大变更
├─ 架构调整
└─ 安全性变更

通知格式：
【待决策项】
- 项目：AI Project Management
- 时间：YYYY-MM-DD HH:mm
- 内容：xxx
- 推荐方案：xxx
- 理由：xxx
```
